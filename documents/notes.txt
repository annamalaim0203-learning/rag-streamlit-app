As per security best practices, any whitelisting activity should be carried out by command. In our case (AIRTEL), we get 
the command from each incident. For dockerd, Command: dockerd--config-file/etc/docker/daemon.json.

We have created a custom rule with strong syntax which will whitelist every single particular incidents. 

Message SAMPLE - 

usr/bin/dockerd wrote a suspicious packed/encrypted binary to /var/lib/docker/vfs/dir/
ffc37059e2677e1104471eabf3a3c76808e4559542929e18400770f2f0a3c606/usr/lib/x86_64-linux-gnu/gconv/GB18030.so. 
Packing/encryption can conceal malicious executables. MD5: 6a92299c277c5f8531ef2ed280445514. 
Command: dockerd--config-file/etc/docker/daemon.json

syntax - file.path startswith "/var/lib/docker/vfs/dir/" and file.path contains "/x86_64-linux-gnu/"
or file.path contains "GB18030.so" or file-path contains "IBM933.so" or file.path
contains "IBM932.so" or file.path contains "IBM930.so"

Breakdown of the synntax - 

1. file.path startswith - Pre defined Syntax from Prisma for type Filesystem (PRESS ctrl + space bar for more options)

2. "/var/lib/docker/vfs/dir/" is the staic path of the incident message.

3. and(AND) operator - multiplication table - when both condition is TRUE, it will be TRUE.

4. "/x86_64-linux-gnu/" - this is the next static path mentioned in the incident message.

5. or(OR) operator - Addition table - If any one of the condition is TRUE, it will be true.

6. "GB18030.so" - Libray which is captured in the incident message. Libraries are invoked to do a particular task.
It can't do any other task out of it.

7. Following this we can add multiple libraries captured in the incident message and map it in the custom rule.
Finally after creating syntax set the effect to ALLOW.

Explanation


The context of Jenkins CI/CD pipelines, a JNLP (Java Network Launch Protocol) container is typically used for managing
agent nodes (also known as build agents or slaves) that run jobs in a distributed Jenkins setup. Here's why a JNLP
container is often needed in Jenkins for CI/CD:
1. Distributed Builds:
Jenkins can operate in a distributed mode where the master (main Jenkins server) manages the orchestration and
scheduling of jobs, while the build tasks themselves are executed on separate agent nodes. These agents can be on
different machines or containers to scale out the CI/CD pipeline.
JNLP containers enable communication between the Jenkins master and these agent nodes. The NLP protocol allows
the master to connect to a remote agent over a network.
2. Containerization:
In modern CI/CD environments, especially in microservices architectures, Jenkins agents are often run inside containers
(e.g., Docker). Using a JNLP container, the Jenkins master can automatically spin up new containers, register them as
Jenkins agents, and offload build tasks to them dynamically.
This helps ensure that the agents are ephemeral, meaning they can be spun up when needed and terminated after the
job is completed, optimizing resource usage
3. Security and Isolation:
Using JNLP containers enhances security by isolating the build process. Instead of running builds on shared or directly
exposed machines, each build job can run inside a secure container. This ensures that any potentially harmful code or
dependencies don't affect other parts of the infrastructure.
Containers also provide a clean environment for every job, ensuring consistent builds and eliminating the "works on my
machine" problem.
4. Cross-Platform Compatibility:
JNLP containers can be used to run Jenkins agents on various platforms (e.g., Linux, Windows, macOs) without needing
to install Jenkins agents directly on each machine. By using JNLP, Jenkins masters can interact with agents running on
different operating systems in a consistent manner
5. Dynamic Scaling:
Containers Can be spun up and down quickly based on demand, enabling Jenkins to dynamically scale the number of
agents available for processing builds. This in particularly useful when running multiple parallel jobs or when worktoads
fluctuate, as it avoids the need to pre-allocate or maintain idle build nodes.
6. Simplifying Setup:
Using a JNLP container eliminates the need to manually configure Jenkins agents. The container can be easily set up with
simple script that connects it to the Jenkins master, reducing manual intervention ầnd complexity when adding new
agents to the Jenkins cluster.
7. Jenkins Agent Docker Image:
Many Jenkins setups use a pre-built Jenkins agent Docker image that includes the necessary JNLP agent capabilities.
When Jenkins needs to launch a new agent, it simply starts a container from this image, which connects to the Jenkins
master via the JNLP protocol and starts running jobs.
In Summary:
A JNLP container is needed in Jenkins CI/CD pipelines to facilitate dynamic, distributed, and scalable builds, It enables
Jenkins to use remote agents (inside containers) to execute jobs, providing isolation, security, and efficient use of
resources in a containerized environment. This approach simplifies the management of build agents, especially in large-
scale or cloud-based CI/CD environments
